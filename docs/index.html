RDF Knowledge Graph for LLMs

hippocamp

A structured brain for any AI agent. Build, query, and persist knowledge graphs — for code, research, or any domain.

brew brew install tggo/tap/hippocamp
01

What is it?

Hippocamp is an MCP server that gives any LLM a persistent, queryable RDF knowledge graph. Named after the brain's memory center — it gives AI agents a structured place to store and recall knowledge.

RDF represents facts as triples: Subject → Predicate → Object. Graphs of triples form a semantic web of knowledge that any LLM can build, query with SPARQL, and persist across sessions.

Works for any domain: code analysis, research notes, API documentation, business processes. The built-in ontology has a generic base layer (topics, entities, notes, decisions) and an optional code layer (files, symbols, dependencies). Extend with your own classes — RDF is open-world.

Backed by BadgerDB in-memory with full named graph support. Data is serialized in TriG format — auto-loaded on startup, auto-saved on shutdown.

02

Four tools

i
triple
add remove list

CRUD operations on individual triples. Add URIs, literals, or blank nodes. List with wildcard pattern matching on subject, predicate, or object. Supports language tags and XSD datatypes.

ii
sparql
SELECT ASK CONSTRUCT UPDATE

Full SPARQL 1.1 query and update engine. Write expressive graph queries, perform pattern matching, aggregations. Update operations auto-detected by keyword (INSERT, DELETE, LOAD...).

iii
graph
create delete list dump load prefix_*

Named graph lifecycle management. Save and load graphs from TriG files. Register namespace prefixes. Get stats on any named graph. Clear graphs without deleting them.

iv
search
keyword type filter scope

Semantic keyword search across the entire graph. Matches labels, summaries, file paths, signatures, and content. Filter by RDF type or named graph scope. Returns ranked results with properties.

03

Examples

// Add a literal with language tag
{
  "action": "add",
  "subject":   "http://example.org/Alice",
  "predicate": "http://example.org/name",
  "object":    "Alice",
  "object_type": "literal",
  "lang": "en"
}

// Add to a named graph
{
  "action": "add",
  "graph":     "http://example.org/people",
  "subject":   "http://example.org/Alice",
  "predicate": "rdf:type",
  "object":    "http://example.org/Person"
}

// List all triples for a subject (wildcard)
{
  "action": "list",
  "subject": "http://example.org/Alice"
}
// SELECT — returns JSON bindings
{
  "query": "SELECT ?s ?name WHERE { ?s <http://example.org/name> ?name } LIMIT 10"
}

// ASK — returns "true" or "false"
{
  "query": "ASK { <http://example.org/Alice> a <http://example.org/Person> }"
}

// INSERT DATA (auto-detected as update)
{
  "query": "INSERT DATA { <http://example.org/Alice> <http://example.org/age> 30 }"
}

// Query a specific named graph
{
  "query": "SELECT ?s WHERE { ?s ?p ?o }",
  "graph": "http://example.org/people"
}
// Create a named graph
{ "action": "create", "name": "http://example.org/people" }

// Save everything to disk
{ "action": "dump", "file": "./backup.trig" }

// Load from file (merges with existing)
{ "action": "load", "file": "./backup.trig" }

// Register a namespace prefix
{ "action": "prefix_add", "prefix": "ex", "uri": "http://example.org/" }

// Stats for a graph
{ "action": "stats", "name": "http://example.org/people" }
// → {"triples": 42}
// Search by keyword
{ "query": "authentication middleware" }

// Filter by type
{
  "query": "Store",
  "type":  "https://hippocamp.dev/ontology#Struct"
}

// Search within a project graph
{
  "query": "database connection",
  "scope": "project:myapp",
  "limit": 5
}

// CLI one-shot query (for hooks/scripts)
// $ hippocamp --query "auth" --type "hippo:Function" --limit 10
# config.yaml
store:
  default_file: "./data/default.trig"
  auto_load: true
  format: "trig"

prefixes:
  ex:     "http://example.org/"
  schema: "http://schema.org/"

---

# Claude Code / Desktop config
{
  "mcpServers": {
    "hippocamp": {
      "command": "hippocamp",
      "args": ["--config", "/path/to/config.yaml"]
    }
  }
}

05

Under the hood

◈
Named graphs

Multiple isolated named graphs in a single store. Each graph is independently queryable. One graph per project, shared ontology.

◉
BadgerDB

Context-aware in-memory store with ACID transactions. Proper graph isolation — no shared state.

◎
TriG format

Persistence in TriG — a Turtle superset that preserves named graph boundaries across sessions.

◌
SPARQL 1.1

Full W3C-conformant query engine. SELECT, ASK, CONSTRUCT, and all UPDATE operations.

◇
Ontology

Built-in hippo: ontology with base layer (topics, entities, notes, decisions) and code layer (files, symbols, dependencies). Open-world extensible.

◆
Claude Code hooks

Pre-query hook loads graph context before each prompt. Post-edit hook tracks changes for re-indexing. Skills for automated project analysis.
